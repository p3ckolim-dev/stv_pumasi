# pumasi v0.1.22 Safe Animal Work Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add safe animal work to Pumasi: pet unpetted animals and collect loose animal-product objects, behind the existing `Animals` work category toggle.

**Architecture:** Keep the current host-authoritative model. The host scans animal houses, queues validated animal tasks, executes world changes through explicit handlers, and broadcasts the existing todo/helper state snapshots. Guests can see/request work but must not mutate animal state. Do not change todo ordering, dedupe, multiplayer authority, or risky planting/chest behavior in this version.

**Tech Stack:** C#/.NET 6, SMAPI 4.x, Stardew Valley 1.6, xUnit, Gemini REST API, Korean/English Pumasi UI strings.

## Scope

This plan targets **v0.1.22** only.

Implement:

- `PetAnimal` scan candidates for unpetted animals in loaded animal houses.
- `CollectAnimalProduct` scan candidates for loose, pickupable animal-product objects in loaded animal houses.
- Execution handlers for `PetAnimal` and safe loose-product collection.
- Collected loose products are stacked into an existing chest stack for the same item when possible.
- `Animals` work category remains the single gate for hay refill, petting, and loose product collection.
- Gemini planning prompt explicitly supports `PetAnimal` and `CollectAnimalProduct` when those tasks appear in `candidateTasks`.
- Localized completion/skip/failure reasons for animal work.
- User/developer docs and settings copy that explain the new animal work and reward/XP behavior.
- Version/package/release updates for `0.1.22`.

Do not implement in v0.1.22:

- Milking cows/goats or shearing sheep through tool-required interactions.
- Truffle collection outside animal houses.
- Any manual XP grant, XP redistribution, or fake helper XP account.
- Guest-owned reward routing.
- New NPC movement/animation behavior.
- Planting, chest sorting, selling, destroying, or inventory-moving automation.
- Creating new chest categories by placing items into arbitrary empty chests.

## Reward And XP Policy

Pumasi is not a real multiplayer farmhand. It executes host-side world changes.

- Rewards and side effects from vanilla action paths belong to `Game1.player`, which is the host on the authoritative execution path.
- Do not call `gainExperience`, `gainFarmerExperience`, or any similar XP API manually in this version.
- Do not split XP or items among guests.
- Do not create a helper XP account.
- If an action is implemented through a vanilla interaction and the game grants XP, friendship, stats, or items, those effects go to the host.
- If an action is implemented by directly changing state, it should not add XP by hand.
- Document this clearly: Pumasi work rewards are host-owned for now.

This means animal petting should update the animal through the game's normal petting path where possible. Loose-product collection uses validated direct storage routing so it can prefer same-item chests before host inventory; it must not grant XP by hand, mutate friendship, or remove the world object unless full storage/inventory transfer succeeds.

## Product Storage Policy

Loose animal products should not disappear into an arbitrary location after collection. Use this conservative routing:

- First, find a normal chest that already contains the exact same stackable item. Match by qualified item ID and quality.
- If several matching chests exist, prefer the closest chest in the same location, then any loaded location with available stack space.
- Add to that matching chest only if it can accept the full product stack. It may merge into an existing stack or create a new same-item stack in an empty slot of that same matching chest.
- If no matching chest can accept the item, add the product to the host inventory.
- If the host inventory cannot accept the item, do not remove the product from the world; skip with a visible reason.
- Do not move unrelated chest items, sort chests, or choose a random empty chest in this version.

This keeps the behavior natural: Pumasi respects the player's existing chest organization, and if it cannot infer a destination, it hands the item to the host instead.

## Acceptance Criteria

- With `Animals` disabled, scan and Gemini candidate summaries do not include `PetAnimal`, `CollectAnimalProduct`, or animal hay refill tasks.
- With `Animals` enabled, loaded animal houses can produce:

```text
#1 [Queued] Pet animal Barn(12,8) P72 source=scan - Unpetted animal
#2 [Queued] Collect animal product Coop(6,4) P82 source=scan - Loose animal product Egg
```

- Animal task keys include animal IDs for petting, so two animals on the same tile cannot dedupe incorrectly.
- Already-petted animals are not queued by scan; if they become petted before execution, the task is skipped with a readable reason.
- Loose animal-product collection revalidates the tile before pickup and skips if the product is gone or is not an allowed loose animal product.
- Loose animal-product collection revalidates that the target location is an animal house before touching objects.
- Tool-required animal products are not collected in this version.
- If inventory cannot accept a loose product, the task is skipped or failed with a readable reason and does not delete the product.
- If a same-item chest stack exists, collected products are stacked there before using host inventory.
- If no same-item chest stack exists, collected products go to host inventory.
- Empty chests are not chosen as new product destinations in this version.
- Pumasi does not manually grant XP. Any vanilla-granted rewards belong to the host.
- `/pms todo` and the board show localized task type and animal result reason through the existing formatter.
- Guests receive synced todo/helper state; guests do not run animal execution locally.
- Tests pass with `./.dotnet/dotnet test Pumasi.sln --no-restore`.

## Task 4: Execute Animal Tasks Safely With Storage Routing

**Files:**
- Modify: `src/Pumasi/Tasks/FarmTaskExecutor.cs`

- [ ] **Step 1: Add switch cases**

Add:

```csharp
TaskType.PetAnimal => PetAnimal(location, task.Target),
TaskType.CollectAnimalProduct => CollectAnimalProduct(location, task.Target),
```

- [ ] **Step 2: Implement `PetAnimal` with revalidation**

Rules:

- Require `target.EntityId`; otherwise skip `animal-id-required`.
- Resolve the animal in the target animal house by ID.
- If not found, skip `animal-not-found`.
- If already petted, skip `animal-already-petted`.
- Invoke the game's normal petting method using `Game1.player` where possible.
- Return complete `petted-animal` only after the animal is actually petted.
- If no safe vanilla method is available, skip or fail with `animal-pet-method-not-found`.
- Do not manually grant XP.
- Do not manually adjust friendship unless that is the game's normal petting path.

Prefer one small helper for method invocation so the risky reflection stays contained.

- [ ] **Step 3: Implement `CollectAnimalProduct` with revalidation**

Rules:

- Re-check that the target location is an animal house.
- Look up the object at `target.X,target.Y`.
- If no object exists, skip `animal-product-not-found`.
- Re-check that the object is an allowed loose animal product.
- If not, skip `not-animal-product`.
- If the host inventory cannot accept it, skip or fail with `inventory-full`.
- Use validated direct storage routing so same-item chests can be preferred before host inventory.
- First try to store it in an existing same-item chest stack.
- If no same-item chest stack can accept it, add it to the host inventory.
- Return complete `stored-animal-product-in-chest` when it went into a chest.
- Return complete `collected-animal-product` when it went into host inventory.
- If neither destination can accept it, skip `inventory-full`.
- Do not manually grant XP.

- [ ] **Step 4: Avoid silent destructive fallbacks**

Do not delete objects directly as a fallback for failed pickup. The product must remain in the world if collection cannot be completed safely.

- [ ] **Step 5: Add same-item chest routing helpers**

Rules:

- Search normal chests in loaded locations.
- A chest is eligible only if it already has an item stack matching the product's qualified item ID and quality.
- Prefer same-location chests before other loaded locations.
- Prefer nearer chests when comparing chests in the same location.
- It is okay to use an empty slot inside that same matching chest, because the chest already has the item category.
- If a matching chest cannot accept the full product stack, continue searching.

Suggested helper names:

```csharp
TryStoreInMatchingChest(GameLocation sourceLocation, Vector2 productTile, SObject product)
TryAddToMatchingChest(Chest chest, Item item)
```
